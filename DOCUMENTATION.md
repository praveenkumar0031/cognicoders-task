# Employee Leave Management System API Documentation

## 1. Project Overview
The **Employee Leave Management System** is a robust RESTful API built with **Node.js** and **Express.js**. It provides a complete backend solution for managing employee records, processing leave requests, and tracking approval workflows.

- **Purpose**: To automate the leave application and approval process within an organization.
- **Tech Stack**: 
  - **Runtime**: Node.js
  - **Framework**: Express.js
  - **Storage**: In-memory array store (no external database required for setup).
- **Key Features**:
  - Comprehensive Employee Management (CRUD-lite).
  - Leave Request Submission with automated duration calculation.
  - Conflict detection (overlap check) for leave requests.
  - Approval/Rejection workflow with mandatory remarks for rejections.
  - Leave summary reports for individual employees.
  - System-wide audit logs for transparency.
- **Base URL**: `http://localhost:3000`

## 2. Project Folder Structure
The project follows a modular architecture based on the **Separation of Concerns** principle.

```text
Cognicoders/
├── app.js              — The main entry point. Configures middleware, routes, and starts 
|                          the server.
├── routes/             — Express routers that define the API endpoints and map them 
|                         to controllers.
├── controllers/        — Logic layer that handles HTTP requests, extracts parameters, 
|                          and sends responses.
├── services/           — Business logic layer. Contains core algorithms like 
|                          overlap checks and duration calculation.
├── validations/        — Schema validation logic to ensure incoming data
|                          meets requirements.
├── middleware/         — Functions that run during the request-response cycle 
|                         (e.g.,   Error Handling, Logging).
├── data/               — The data persistence layer (in-memory store for this project).
└── package.json        — Project metadata, scripts, and dependencies.
```

### Responsibility Flow:
1. **Route**: Receives the request and directs it to the appropriate Controller.
2. **Controller**: Validates input using the Validation layer and passes clean data to the Service.
3. **Service**: Performs business logic and interacts with the Data Store.
4. **Data Store**: Performs CRUD operations on the in-memory arrays.
5. **Response**: The Controller formats the service output into a unified JSON response.

## 3. Architecture & Data Flow

### Request Lifecycle
When a client makes a request (e.g., `POST /api/leaves`):
1. **Routing**: `app.js` routes the request to `leaveRoutes.js`.
2. **Controller**: `leaveController.js` receives the request body.
3. **Validation**: The controller calls `leaveValidation.js`. If data is invalid, it immediately returns a `400 Bad Request`.
4. **Service**: The controller calls `leaveService.js`. The service checks if the employee exists, checks for date overlaps, and calculates the duration.
5. **Data Store**: The service saves the new leave object into the `leaves` array in `store.js`.
6. **Audit Log**: The service triggers `auditService.js` to record the action.
7. **Response**: The controller sends a `201 Created` success response.

### Global Error Handling
The system uses a centralized error-handling middleware (`middleware/errorHandler.js`). Any error thrown in the services or controllers is caught here, logged, and returned to the user in a consistent format:
```json
{
  "success": false,
  "message": "Error description here"
}
```

## 4. Data Models

### Employee Object
Represents a staff member in the organization.
| Field | Type | Description | Example |
| :--- | :--- | :--- | :--- |
| `employeeId` | String | Unique identifier (e.g., E001) | "E001" |
| `name` | String | Full name of the employee | "Rahul Sharma" |
| `email` | String | Official email address | "rahul@example.com" |
| `department` | String | Employee's department | "Engineering" |
| `joiningDate` | String | Date of joining (YYYY-MM-DD) | "2023-01-15" |
| `createdAt` | String | Timestamp of record creation | "2024-06-01T10:00:00.000Z" |

### Leave Object
Represents a leave application.
| Field | Type | Description | Example |
| :--- | :--- | :--- | :--- |
| `leaveId` | String | Unique UUID generated by system | "l8-x9-..." |
| `employeeId` | String | Reference to the Employee | "E001" |
| `leaveType` | String | Type of leave (Sick, Casual, etc.) | "Sick" |
| `startDate` | String | Leave start date (YYYY-MM-DD) | "2025-06-10" |
| `endDate` | String | Leave end date (YYYY-MM-DD) | "2025-06-12" |
| `duration` | Number | Calculated days (inclusive) | 3 |
| `reason` | String | Reason for leave | "Fever" |
| `status` | String | Pending, Approved, or Rejected | "Pending" |
| `remarks` | String | Rejection comments (mandatory if rejected) | null |
| `createdAt` | String | Timestamp of application | "2025-06-01T10:00:00.000Z" |

### Audit Log Object
Immutable records of system actions.
| Field | Type | Description | Example |
| :--- | :--- | :--- | :--- |
| `action` | String | The event name | "Leave Created" |
| `leaveId` | String | Related Leave ID | "L001" |
| `employeeId` | String | Related Employee ID | "E001" |
| `remarks` | String | Additional context (e.g., rejection reason) | "Insufficient details" |
| `timestamp` | String | When the event occurred | "2025-06-01T10:05:00.000Z" |

## 5. API Reference

### Employee Endpoints

#### POST `/api/employees`
**Description**: Onboard a new employee.
- **Body**:
  - `employeeId` (String, Required): Unique ID.
  - `name` (String, Required): Min 3 chars.
  - `email` (String, Required): Valid format.
  - `department` (String, Required).
  - `joiningDate` (String, Required): YYYY-MM-DD.
- **Success Response (201)**:
```json
{
  "success": true,
  "message": "Employee created successfully",
  "data": { "employeeId": "E001", "name": "Rahul Sharma", ... }
}
```

#### GET `/api/employees`
**Description**: Retrieve all employees with optional filters.
- **Query Params**:
  - `department` (String): Filter by department.
  - `search` (String): Search in name or email.
- **Example**: `GET /api/employees?department=Engineering&search=rahul`
- **Success Response (200)**:
```json
{
  "success": true,
  "data": [ { "employeeId": "E001", "name": "Rahul Sharma", ... } ]
}
```

---

### Leave Endpoints

#### POST `/api/leaves`
**Description**: Submit a new leave request.
- **Body**:
  - `employeeId` (String, Required).
  - `leaveType` (String, Required): e.g., "Sick", "Casual".
  - `startDate` (String, Required).
  - `endDate` (String, Required).
  - `reason` (String, Required).
- **Success Response (201)**:
```json
{
  "success": true,
  "message": "Leave request created successfully",
  "data": { "leaveId": "...", "duration": 3, "status": "Pending", ... }
}
```

#### GET `/api/leaves`
**Description**: Fetch leave requests with advanced filtering, sorting, and pagination.
- **Query Params**:
  - `employeeId` (String).
  - `status` (String): Pending, Approved, Rejected.
  - `page` (Number): Default 1.
  - `limit` (Number): Default 10.
  - `sortBy` (String): Field name (e.g., startDate).
  - `order` (String): asc or desc.
- **Success Response (200)**:
```json
{
  "success": true,
  "total": 15,
  "page": 1,
  "limit": 5,
  "data": [ ... ]
}
```

---

### Approval Endpoints

#### PATCH `/api/leaves/:leaveId/approve`
**Description**: Approve a pending leave.
- **Success Response (200)**:
```json
{
  "success": true,
  "message": "Leave approved successfully",
  "data": { "status": "Approved", ... }
}
```

#### PATCH `/api/leaves/:leaveId/reject`
**Description**: Reject a pending leave.
- **Body**:
  - `remarks` (String, Required): Reason for rejection.
- **Success Response (200)**:
```json
{
  "success": true,
  "message": "Leave rejected successfully",
  "data": { "status": "Rejected", "remarks": "Need more info", ... }
}
```

---

### Summary & Logs

#### GET `/api/leaves/summary/:employeeId`
**Description**: Get leave statistics for a specific employee.
- **Success Response (200)**:
```json
{
  "success": true,
  "data": {
    "employeeId": "E001",
    "employeeName": "Rahul Sharma",
    "totalLeaveRequests": 5,
    "approved": 3,
    "pending": 1,
    "rejected": 1
  }

```

#### GET `/api/audit-logs`
**Description**: View all system activity logs (Newest first).
- **Success Response (200)**:
```json
{
  "success": true,
  "data": [ { "action": "Leave Approved", "timestamp": "...", ... } ]
}
```

| `email` | Employee | Valid Email Regex | "Valid email is required" |
| `department` | Employee | Mandatory | "Department is required" |
| `joiningDate` | Employee | Not in Future | "Joining date cannot be in the future" |
| `startDate` | Leave | Mandatory | "Start date is required" |
| `endDate` | Leave | Not before `startDate` | "End date cannot be before start date" |
| `duration` | Leave | Max 10 days | "Leave duration cannot exceed 10 days" |
| `overlap` | Leave | No overlap with Pending/Approved | "Leave request overlaps with an existing..." |
| `remarks` | Rejection | Mandatory | "Remarks are mandatory for rejection" |

## 7. Error Handling

All errors return a non-2xx status code and follow this structure:
```json
{
  "success": false,
  "message": "Descriptive error message here"
}
```

### Standard HTTP Status Codes:
- **200 OK**: Request successful.
- **201 Created**: Resource successfully created.
- **400 Bad Request**: Validation failure or business logic violation (e.g., overlap).
- **404 Not Found**: Resource (Employee/Leave) does not exist.
- **500 Internal Server Error**: Unexpected server-side failure.

## 8. Sample Test Scenarios

### Scenario 1 — Happy Path

**Step 1: Create Employee E001 (Rahul Sharma)**
- **URL**: `POST /api/employees`
- **Body**:
```json
{
  "employeeId": "E001",
  "name": "Rahul Sharma",
  "email": "rahul@example.com",
  "department": "Engineering",
  "joiningDate": "2023-01-15"
}
```
- **Response**: `201 Created`

**Step 2: Create Employee E002 (Priya Mehta)**
- **URL**: `POST /api/employees`
- **Body**:
```json
{
  "employeeId": "E002",
  "name": "Priya Mehta",
  "email": "priya@example.com",
  "department": "HR",
  "joiningDate": "2023-05-20"
}
```
- **Response**: `201 Created`

**Step 3: Submit Leave Request for Rahul**
- **URL**: `POST /api/leaves`
- **Body**:
```json
{
  "employeeId": "E001",
  "leaveType": "Sick",
  "startDate": "2025-07-01",
  "endDate": "2025-07-03",
  "reason": "Viral Fever"
}
```
- **Response**: `201 Created` (Duration: 3)

**Step 4: Submit Another Leave for Rahul**
- **URL**: `POST /api/leaves`
- **Body**:
```json
{
  "employeeId": "E001",
  "leaveType": "Casual",
  "startDate": "2025-07-10",
  "endDate": "2025-07-14",
  "reason": "Family Function"
}
```
- **Response**: `201 Created` (Duration: 5)

**Step 5: Approve First Leave**
- **URL**: `PATCH /api/leaves/{{leaveId_from_step3}}/approve`
- **Response**: `200 OK` (Status: "Approved")

**Step 6: Reject Second Leave**
- **URL**: `PATCH /api/leaves/{{leaveId_from_step4}}/reject`
- **Body**: `{ "remarks": "Project deadline on those dates" }`
- **Response**: `200 OK` (Status: "Rejected")

**Step 7: Check Rahul's Summary**
- **URL**: `GET /api/leaves/summary/E001`
- **Response**: `200 OK`
```json
{
  "success": true,
  "data": {
    "employeeId": "E001",
    "employeeName": "Rahul Sharma",
    "totalLeaveRequests": 2,
    "approved": 1,
    "pending": 0,
    "rejected": 1
  }
}
```

### Scenario 2 — Validation Failures

**Step 1: Duplicate Employee ID**
- **Request**: Create employee with `employeeId: E001` again.
- **Response**: `400 Bad Request`
- **Message**: "Employee ID already exists"

**Step 2: Invalid Email**
- **Request**: `email: "rahul.com"`
- **Response**: `400 Bad Request`
- **Message**: "Valid email is required"

**Step 3: Overlapping Leave**
- **Request**: Submit leave for E001 from `2025-07-02` to `2025-07-04` (overlaps with approved leave in Scenario 1).
- **Response**: `400 Bad Request`
- **Message**: "Leave request overlaps with an existing pending or approved leave"

**Step 4: Rejecting without Remarks**
- **Request**: `PATCH /api/leaves/{{id}}/reject` with empty body.
- **Response**: `400 Bad Request`
- **Message**: "Remarks are mandatory for rejection"

## 10. Setup & Running the Project

### Prerequisites
- **Node.js**: v14.x or higher
- **npm**: v6.x or higher

### Installation Steps
1. Clone the repository or navigate to the project folder.
2. Install dependencies:
   ```bash
   npm install
   ```

### Running the Server
Start the API server:
```bash
npm start
```
The server will start at `http://localhost:3000`. You should see `Server is running on port 3000` in the terminal.

### Verification
Open your browser or a tool like **Postman** and visit:
`http://localhost:3000/`
You should receive:
```json
{
  "success": true,
  "message": "Employee Leave Management System API is running"
}
```

### Recommended Testing Tools
- **Postman**: Best for collections and environment variables.
- **Thunder Client**: VS Code extension for quick testing.
- **curl**: For command-line testing.
